1. 전체 구조 이해

Harness-1은 강화학습으로 학습된 20B 검색 에이전트입니다. 핵심은 모델을 코드에 직접 로드하는 구조가 아니라, 별도로 서빙되는 모델 서버에 HTTP로 붙는 클라이언트 구조라는 점입니다. 에이전트 루프·검색 도구·상태 관리는 클라이언트(이 코드베이스)에 있고, 토큰 생성만 모델 서버(vLLM)에 위임합니다.

그림 2 · 시스템 아키텍처 — 에이전트(클라이언트) ↔ 모델 서버 ↔ 검색 백엔드
ℹ️
3가지 필수 요소 — 에이전트를 의미 있게 돌리려면 ①모델 서버, ②검색 백엔드(코퍼스/벡터스토어), ③동작 플래그(V8D_*)가 모두 있어야 합니다. 모델만 있고 검색 백엔드가 없으면 첫 검색에서 실패합니다.

1 Qdrant 어댑터 도구 작성

기본 검색 도구는 harness/tools.py에서 Chroma 네이티브 API (chromadb.Search / Rrf / Knn / Key.DOCUMENT.regex)에 강결합돼 있습니다. 따라서 Qdrant 컬렉션을 "그대로 꽂을" 수 없고, 아래 3개 도구를 Qdrant API로 재구현해야 합니다.

그림 4 · 하이브리드 검색(Dense+Sparse RRF) — Chroma·Qdrant 양쪽에서 동일 패턴으로 매핑 가능
도구Chroma 기존Qdrant 대응
SearchCorpusToolRrf(Knn dense + Knn bm25)Query API prefetch + fusion=RRF
GrepCorpusToolKey.DOCUMENT.regex()full-text match (⚠️ 정규식 제약)
ReadDocumentToolID로 getretrieve by ID
클라이언트config.get_chroma_client()QdrantClient(url, api_key)
🚧
grep(정규식)이 가장 까다로움 — Chroma는 문서 텍스트에 임의 정규식을 걸지만, Qdrant는 payload full-text 부분일치/키워드 match는 되어도 임의 정규식은 네이티브 지원이 약합니다. full-text 인덱스 + match로 우회하거나 scroll+필터로 처리해야 합니다.

config.py 추가 예시

# harness/config.py
from qdrant_client import QdrantClient

def get_qdrant_client(self) -> QdrantClient:
    return QdrantClient(
        url=self.qdrant_url.get_secret_value(),
        api_key=self.qdrant_api_key.get_secret_value(),
    )

※ QDRANT_URL, QDRANT_API_KEY, QDRANT_COLLECTION을 .env.local에 추가.

SearchCorpusTool (Qdrant) 핵심 — RRF 하이브리드

from qdrant_client import models

def __call__(self, params, overrides=None):
    query = params["query"]
    dense = self.create_embeddings([query])[0]      # ← [2]단계에서 모델 맞춤

    res = self._client.query_points(
        collection_name=self._collection,
        prefetch=[
            models.Prefetch(query=dense, using="dense_vector", limit=self._knn_limit),
            # 희소(BM25) 벡터를 색인했다면 sparse prefetch 추가 가능
        ],
        query=models.FusionQuery(fusion=models.Fusion.RRF),
        limit=self._search_limit,
        with_payload=True,
    ).points

    ids  = [p.id for p in res]
    docs = [p.payload["text"] for p in res]          # payload 필드명은 색인 스키마에 맞춤
    # ... 리랭커 / 포맷팅은 기존 로직 재사용 ...
ℹ️
반환 형식((formatted_string, metadata) 튜플)과 returned_chunk_ids 메타데이터는 기존 Chroma 버전과 동일하게 맞춰야 에이전트 루프·큐레이션이 그대로 동작합니다.

2 임베딩 모델 교체

text-embedding-3-small은 고정이 아니라 harness/tools.py:485의 create_embeddings에 하드코딩돼 있을 뿐입니다(원작자도 "TODO: 필드로 빼자"고 주석). 한 줄 수정으로 변경 가능합니다.

그림 5 · 임베딩을 바꿔도 되지만, 쿼리와 색인이 같은 모델이어야 검색이 성립
⚠️
색인이 다른 모델이면 → 코퍼스 재임베딩 대신 쿼리 쪽을 색인 모델에 맞추는 것이 훨씬 저렴합니다(한 줄 수정).

케이스별 대응

Qdrant 색인 모델해야 할 일
text-embedding-3-small (1536)변경 불필요
다른 OpenAI 모델 (예: 3-large 3072)485줄 model 문자열만 교체 + 차원 확인
로컬/Ollama (예: nomic-embed-text 768)base_url을 로컬 임베딩 서버로 + model명 교체
완전 다른 SDKcreate_embeddings 본문 자체를 교체

수정 예시 — 로컬/Ollama 임베딩으로

# harness/tools.py  (create_embeddings)
def create_embeddings(self, texts):
    resp = self._openai_client.embeddings.create(
        model=self._openai_ef_name,   # 하드코딩 대신 필드 사용
        input=texts,
        encoding_format="float",
    )
    return [e.embedding for e in resp.data]

# 생성 시:
#   openai_client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
#   SearchCorpusTool(..., openai_ef_name="nomic-embed-text")
ℹ️
차원·하이브리드 주의 ① dense 벡터 차원이 새 모델 출력과 일치(3-small=1536 / 3-large=3072 / nomic=768). ② BM25 sparse는 쿼리 시 로컬 계산이라 임베딩 교체와 무관. ③ text-embedding-3-small을 전체 검색해 다른 도구(ReadDocumentTool 등)도 함께 교체.
🎯
벤치마크 충실도 — harness-1 모델은 검색된 텍스트를 읽지 임베딩을 보지 않으므로, 임베딩을 바꿔도 에이전트는 정상 동작합니다. 단 검색 품질(recall)이 달라지니, 논문 수치 재현이 목적이면 text-embedding-3-small 유지가 공정합니다.
